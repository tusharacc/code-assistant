REQUIREMENT DOCUMENT
Project: CLI Log Analyser and Alerting Tool
Version: 1.0
Purpose: Benchmarking code assistant tools
=========================================


OVERVIEW
--------
Build a command-line tool called  logscope  that parses structured log files
(JSON-lines or a configurable regex pattern), aggregates metrics, detects
anomalies, and outputs reports as plain text, JSON, or HTML. It must handle
files up to several GB without loading the entire file into memory.

This benchmark tests: file streaming, regex, data aggregation, configurable
rule evaluation, multiple output formats, a plugin system, and a non-trivial
CLI with subcommands.


FUNCTIONAL REQUIREMENTS
-----------------------

1. INPUT FORMATS
   1.1  JSON-lines (JSONL): one JSON object per line. Each object must contain
        at least a timestamp field and a level field.
   1.2  Plain-text with a configurable regex pattern supplied via --pattern flag.
        The pattern must define named groups: timestamp, level, message, and
        optionally service and request_id.
   1.3  Gzip-compressed files (*.gz) must be decompressed transparently.
   1.4  Log lines that fail to parse are counted and reported but do not abort
        processing.

2. SUBCOMMANDS
   The tool exposes three subcommands:

   2a. SUMMARY
       logscope summary [OPTIONS] <file> [<file> ...]

       Aggregates and prints:
         - Total lines parsed / lines skipped (parse failures)
         - Time range: earliest and latest timestamp found
         - Log level breakdown (count + percentage per level: DEBUG, INFO,
           WARN, ERROR, FATAL or equivalent)
         - Top 10 most frequent messages (by exact text, ignoring timestamps
           and IDs stripped by a normaliser — see requirement 6)
         - Error rate over time: split the time range into N equal buckets
           (default 10, configurable via --buckets) and show the error count
           per bucket as a sparkline or ASCII bar chart.

   2b. FILTER
       logscope filter [OPTIONS] <file> [<file> ...]

       Streams matching lines to stdout (or a file via --out).
       Filter options (combinable with AND logic):
         --level    ERROR,WARN        comma-separated list of levels to include
         --service  auth,api          filter by service name (if present)
         --from     2024-01-15T08:00  lower timestamp bound (inclusive)
         --to       2024-01-15T09:00  upper timestamp bound (inclusive)
         --grep     "connection refused"  substring or regex match on message
         --request  abc123            match by request_id (if present)
       Output format: same as input (JSONL → JSONL, plain → plain).

   2c. ALERT
       logscope alert [OPTIONS] <file> [<file> ...]

       Evaluates a set of alert rules (from a YAML/TOML config file specified
       via --rules <path>). Reports which rules fired, with evidence lines.

       Rule schema (YAML example):
         rules:
           - name: "High error rate"
             condition: error_rate_per_minute > 10
             window: 60s
             severity: critical

           - name: "OOM keyword"
             condition: message contains "out of memory"
             severity: warning

           - name: "Slow request"
             condition: duration_ms > 5000
             severity: warning

       Built-in condition types:
         error_rate_per_minute > N   — rolling window error rate
         message contains "text"     — substring match (case-insensitive)
         message matches "regex"     — regex match
         field <name> > N            — numeric field threshold
         field <name> == "value"     — string equality on any JSONL field

       Output for each fired rule:
         [CRITICAL] High error rate — fired at 2024-01-15T08:42:00 (15 errors/min)
         Evidence: <up to 3 matching log lines>

3. OUTPUT FORMATS
   3.1  --format text   (default) — human-readable plain text
   3.2  --format json   — machine-readable JSON, one object per subcommand
        result, written to stdout or --out file
   3.3  --format html   — self-contained HTML file with inline CSS; suitable
        for opening in a browser. Charts use ASCII rendered as <pre> blocks —
        no external JS libraries.

4. STREAMING AND MEMORY
   4.1  The tool must process files line by line; the full file must never be
        held in memory simultaneously.
   4.2  Memory usage must stay below 100 MB regardless of input file size
        (enforced by design — no requirement to measure at test time).
   4.3  Processing speed target (informational, not tested): at least 50 000
        lines/second on a modern single core.

5. MULTIPLE FILE SUPPORT
   5.1  All subcommands accept one or more file paths.
   5.2  Files are processed in the order given; metrics are aggregated across
        all files as if they were a single stream.
   5.3  Standard input is supported when  -  is given as the file argument,
        e.g.  cat app.log | logscope summary -

6. MESSAGE NORMALISER
   6.1  Before counting "top messages", strip variable parts:
          - UUIDs / GUIDs  → {uuid}
          - IPv4 addresses → {ip}
          - Integers       → {int}
          - ISO timestamps embedded in the message → {ts}
   6.2  The normaliser must be a separate, importable module so tests can
        exercise it independently.
   6.3  Normalisation is applied only for aggregation — the raw message is
        always preserved in filter output.

7. PLUGIN SYSTEM
   7.1  A plugin is a Python file placed in a directory specified by the
        environment variable  LOGSCOPE_PLUGIN_DIR  (default: ~/.logscope/plugins).
   7.2  A plugin file exports a function:
          def register(registry): ...
        which calls  registry.add_condition(name, evaluator_func)  to add
        custom alert condition types.
   7.3  Plugins are loaded automatically at startup if the directory exists.
   7.4  A broken plugin (import error) logs a warning but does not abort the tool.

8. CONFIGURATION FILE
   8.1  If a file  .logscope.toml  exists in the current directory or any
        parent directory, it is loaded as default configuration.
   8.2  Supported keys:
          default_format = "text"
          default_buckets = 10
          timestamp_field = "ts"     # JSONL field name for timestamp
          level_field = "level"      # JSONL field name for log level
   8.3  CLI flags override config file values; config file values override
        built-in defaults.


NON-FUNCTIONAL REQUIREMENTS
---------------------------

9.  PROJECT STRUCTURE
      logscope/
        __init__.py
        cli.py           subcommand definitions (Click or Typer)
        parser.py        log line parsing (JSONL + regex modes)
        aggregator.py    metric accumulation (streaming)
        alerting.py      rule loading and evaluation
        normaliser.py    message normalisation (requirement 6)
        renderer.py      text / JSON / HTML output
        plugins.py       plugin loader
      tests/
        test_parser.py
        test_aggregator.py
        test_normaliser.py
        test_alerting.py
        test_cli.py      end-to-end tests using Click's test runner

10. ERROR HANDLING
    10.1 Missing input file → exit code 2, descriptive message.
    10.2 Malformed rules file → exit code 3, descriptive message pointing to
         the offending rule.
    10.3 Unrecognised CLI flag → standard Click/Typer help message.

11. TESTS
    11.1 All test files in tests/ must pass with  pytest  and no network access.
    11.2 Test coverage must include:
           - JSONL parsing (valid line, missing field, malformed JSON)
           - Regex pattern parsing (match, no match, missing named group)
           - Gzip file decompression
           - Message normalisation (UUID, IP, integer stripping)
           - Summary aggregation: correct level counts and time range
           - Filter: level, --from/--to, --grep (combined)
           - Alert: each built-in condition type fires when threshold exceeded
           - Alert: no false positives when threshold is not exceeded
           - HTML output contains no external script/link tags
    11.3 Tests must use small synthetic log fixtures embedded in the test files
         (no external fixture files required).

12. DEPENDENCIES
    12.1 click or typer for CLI
    12.2 pyyaml or tomllib (stdlib in 3.11+) for rule files
    12.3 No external libraries for parsing, aggregation, or rendering
    12.4 pytest for tests
    12.5 A pyproject.toml or setup.cfg must declare all dependencies.


ACCEPTANCE CRITERIA
-------------------

AC-1  logscope summary app.log  prints a level breakdown and time range.
AC-2  logscope summary --format json app.log  outputs valid JSON.
AC-3  logscope summary --format html app.log > report.html  produces a file
      that opens in a browser with no external resource requests.
AC-4  logscope filter --level ERROR --from 2024-01-15T08:00 app.log  outputs
      only ERROR lines within the time range, in the original format.
AC-5  logscope alert --rules rules.yaml app.log  reports fired rules with
      evidence lines and exits 0 if any rules fired, 0 if none.
AC-6  cat large.log.gz | logscope summary -  processes gzip input from stdin
      and prints a summary without error.
AC-7  Placing a plugin file in LOGSCOPE_PLUGIN_DIR and defining a custom
      condition allows that condition to be used in a rules YAML file.
AC-8  logscope summary  on a 500 MB log file runs to completion without an
      out-of-memory error (manual spot-check; not automated).
AC-9  All tests in tests/ pass with  pytest.
